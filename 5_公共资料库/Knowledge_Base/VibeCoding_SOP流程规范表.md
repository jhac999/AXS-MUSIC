# 🚀 VibeCoding 标准协作开发 SOP (Standard Operating Procedure)

> 本文档为 `VibeCoding_全栈开发架构师` 的核心执行法则与工作流规范字典。在任何 AI 辅助开发项目中，必须强制遵循此四步流程，以确保代码质量、杜绝幻觉和“屎山”。

## 📊 核心流程执行表

| 阶段 | 核心任务 | 负责角色 | 输入物 (Input) | 输出物 (Output) | 执行动作与原则 | 硬性验收/约束标准 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Step 1**<br/>`需求收敛` | **锚定开发目标**<br/>明确意图，消除模糊 | 架构师 / 主控 Agent | 用户口语化的想法描述、现有工程环境 | `proposal.md`<br/>(标准需求文档) | 1. **严禁瞎猜**：针对模糊边界、业务逻辑连续向用户发问。<br/>2. 整理项目目标、运行环境、输入/输出路径、核心特性列表。 | 必须得到用户对 `proposal.md` 的明确点头同意，方可进入下一步。 |
| **Step 2**<br/>`架构设计` | **模块化解耦**<br/>宏观与微观设计 | 架构师 / 主控 Agent | `proposal.md` | `detailed-design.md`<br/>(系统设计图纸) | 1. 拆解核心模块（如：UI 层、数据处理层、核心逻辑层）。<br/>2. 设计各模块之间的调用接口与数据结构。<br/>3. 保持“高内聚、低耦合”。 | 模块之间必须解耦；不能有“牵一发而动全身”的超大单体类或函数。 |
| **Step 3**<br/>`任务切分` | **原子化任务生成**<br/>防代码崩塌 | 任务拆解师 / 主控 Agent | `detailed-design.md` | 1. `各模块清单`<br/>2. `progress.md`<br/>(全局进度表) | 1. 为每个大模块拆分成可以直接编码的**单一小任务**。<br/>2. 在 `progress.md` 中使用 Checklist (`[ ]`) 统筹全局任务进度。 | 单个任务的上下文要求必须极度简短，绝不能要求一次性生成整个系统。 |
| **Step 4**<br/>`多Agent编码与监工` | **死磕代码质量**<br/>编码与测试闭环 | **包工头(监工)** +<br/>若干**子 Agent(码农)** | `progress.md` 上的单一小任务 | 1. `生产环境代码`<br/>2. `单元测试代码` | 1. **监工分配任务**：每次唤醒一个上下文纯净的子 Agent 只写一个功能。<br/>2. **强制测试**：要求子 Agent 必须顺带写好相应的单元测试和异常捕获。 | 1. 必须跑通**单元测试**（如 pytest）。<br/>2. 必须跑通**静态语法检测**（如 Python 下的 mypy/ruff）。<br/>3. 只有 0 Error 才能勾选 `[x]`。 |

---

## 🎯 最佳实践指南

1. **绝对隔离原则**：在 `Step 4` 阶段，千万不要用同一个连续对话窗口让 AI 一直写。每一个模块的代码生成，都应该开启一个全新的、“上下文干净”的对话（子 Agent），只给它当前模块需要的 `detailed-design.md` 截断信息。
2. **测试驱动开发 (TDD) 倾向**：包工头在监工时，可以要求子 Agent “先写测试，再写逻辑”，用测试代码来反推并锁死业务代码的边界。
3. **灵活动态**：此 SOP 表虽然以 Python/FC 模拟器为理论蓝本，但在前端交互（HTML/TS）、音频批量处理脚本（Shell/Python）、哪怕是写一套庞大的音乐分发爬虫时，均原样适用！
